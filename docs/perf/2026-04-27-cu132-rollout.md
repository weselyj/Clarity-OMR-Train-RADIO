`cpu_pipeline_pct` = (75 + 106 + 5 + 0.15 + 0.30) / 5180 = **3.6%** — informs Phase 6 decision below.

### Phase 6 — cu132 + DataLoader rewrite (no compile)

Source: `bench/cu132_dataloader/profile.jsonl`, `--warm 100` (250 measured opt-steps after 100 warm). Built with the new `StageBDataset` + `WeightedRandomSampler` pipeline + `pin_memory=True` + `persistent_workers=True` + `non_blocking=True` H2D — `--num-workers 4` default.

| Phase | Mean ms | % of wall | vs Phase 3 |
|---|---|---|---|

| gpu_grad_diagnostics | 117 | 2.3% | — |
| cpu_h2d | 0.86 | 0.0% | **−84%** (non_blocking + pin_memory) |
| cpu_sample | 1.74 | 0.0% | DataLoader iterator overhead replaces inline sampler |

DataLoader workers fully mask cpu_encode + cpu_augment behind GPU compute. The serial-wall savings are 185 ms (encode + augment + h2d) but actual wall delta is 155 ms — the workers have non-zero overhead from queue serialization + worker ramp-up, but a clear net win.

## Summary table

| Configuration | Wall (ms/opt-step) | gpu_fwd (ms) | gpu_bwd (ms) | Δ vs cu128 baseline | Marginal |
|---|---|---|---|---|---|
| **cu128 reference** (issue #2 pinned comment) | 5362 | 1828 | 3255 | — | — |
| **cu132 wheel only** (Phase 3) | 5180 | 1748 | 3097 | **−3.4%** | cu132 wheel: −182 ms |
| **cu132 + DataLoader** (Phase 6) | 5025 | 1760 | 3116 | **−6.3%** | DataLoader: −155 ms (−3.0%) |
| **cu132 + DataLoader + torch.compile** (Phase 4.2) | 4991 | 1746 | 3094 | **−6.9%** | torch.compile: −34 ms (−0.7%) |

Three independent wins compose cleanly:
1. **cu132 wheel** saves ~3.4% on encoder forward + backward (newer cuDNN / kernels).
2. **DataLoader rewrite** saves another ~3.0% by hiding cpu_encode + cpu_augment behind GPU compute.
3. **torch.compile** on decoder + bridge saves an additional ~0.7%.

GPU forward + backward is essentially constant across all three (~4845 ms ±0.7% — noise). Every win is from the input pipeline or kernel-level toolchain improvements; none are from the decoder/bridge model code itself (which is what compile touches), confirming the encoder is the dominant cost on this workload.

## Phase 6 decision (DataLoader gate, retroactively)

The plan's gate said: skip Phase 6 if `cpu_pipeline_pct < 5%`. Phase 3's reading was 3.6%, which by that rule meant SKIP. **Maintainer overrode this gate** during the session (Stage 3 downtime), so Phase 6 was implemented and benched anyway. The actual measured DataLoader contribution turned out to be **−3.0%** wall — about the same magnitude as the cu132 wheel itself. The gate was misleading because `cpu_pipeline_pct` measured serial wall fraction, but the DataLoader's actual win comes from masking via worker prefetch, not from raw CPU savings. Lesson: the gate should be re-formulated as "expected wall improvement from full overlap" rather than "current cpu pipeline %", which is harder to estimate but more accurate.

## Decision: KEEP-OPT-IN

Best cu132 config (Phase 4.2) is **−6.9% wall** vs the cu128 reference, which lands in the 3–10% zone of the plan's rollout matrix → **KEEP-OPT-IN**.

Concretely:
- **`venv-cu132/` is preserved alongside `venv/`** on the Windows host. New runs should prefer `venv-cu132` for the −3.4% wheel-only win.
- **The DataLoader rewrite + `--torch-compile` flag merge to `main` regardless of the venv decision** — they're orthogonal code changes that work on either toolchain. Stage launchers can opt into the −0.7% compile win by adding `--torch-compile`.
- **No tag rename** of the existing venv. The user can switch between the two by editing the launcher's `$venv` path (or by using the existing `setup_venv_cu132.ps1` script to recreate the cu132 venv on a different host).
- **Existing cu128 checkpoints (Stage 1, Stage 2) remain compatible** under cu132 — Phase 3 + 6 + 4.2 all resumed cleanly from the Stage 2 `_final.pt`.

The cu132 win is real but not transformative. KEEP-OPT-IN preserves rollback capability cheaply: if a future cu132 nightly regresses, drop `venv-cu132/` and re-run from `venv/` with no code changes needed.
